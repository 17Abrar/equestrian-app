## Fixes Applied

### 1. Webhook receivers now sanitize unexpected top-level failures

**Files:**
- `apps/web/app/api/webhooks/clerk/route.ts`
- `apps/web/app/api/webhooks/ziina-platform/route.ts`

**Issue:** Stripe, per-club Ziina, and N-Genius webhooks had top-level wrappers that catch unexpected failures from body reads, rate-limit checks, route params, DB claims, or other pre-processing. Clerk and platform-Ziina did not. An unexpected throw outside their inner processing `try/catch` could bubble to the runtime instead of returning the same static, sanitized `Internal error` shape used elsewhere.

**Fix:** Added wrapper `POST` handlers that delegate to internal `handlePost` functions, log sanitized error metadata server-side, and return static `500 Internal error` responses.

### 2. Payment webhook reconciliation failures no longer disappear as processed

**File:** `apps/web/lib/payments/webhook-helpers.ts`

**Issue:** Booking and livery webhook helpers correctly refused to apply paid events when amount/currency data was missing, mismatched, or underfunded, but returned a plain `matched` result. The receiver routes then marked the dedup row `processed`, so operators had to notice logs manually. Platform subscription webhooks already escalated equivalent failures to `permanently_failed`.

**Fix:** These branches now return `permanentFailureReason`, so Stripe, Ziina, and N-Genius receivers park the webhook event as `permanently_failed` and trigger the existing operator-alert path instead of silently consuming the event.

### 3. Livery billing anchor pre-join is explicitly tenant-bound

**File:** `packages/db/src/queries/livery-invoices.ts`

**Issue:** `findHorsesDueForBilling` pre-joined the latest invoice anchor by `horse_id` only. Horse IDs are UUID primary keys, so the current runtime collision risk is low, but the project rule is explicit: tenant-scoped queries must carry `club_id` defensively.

**Fix:** The anchor subquery now selects/groups by `club_id` and joins on both `(horse_id, club_id)`.
