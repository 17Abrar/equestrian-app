## CRITICAL — confirmed authorization-bypass window

### 1. Deactivated admins kept admin powers for the lifetime of their Clerk session

**File:** `apps/web/lib/tenant.ts:67-115` (Path 1 — Clerk active org).

**What was wrong:** when an admin was deactivated via `DELETE /api/v1/staff/[memberId]` (or any deactivate path), `deactivateMember` (`packages/db/src/queries/club-members.ts:291`) flipped `is_active=false` in the DB but did NOT call Clerk's `removeOrganizationMembership`. The deactivated admin's browser session still carried `org_role: 'org:admin'` in the JWT.

`getTenantContext` then ran the orgId resolution path:

```ts
const member = await db
  .select()
  .from(clubMembers)
  .where(
    and(
      eq(clubMembers.clubId, foundClub.id),
      eq(clubMembers.clerkUserId, userId),
      eq(clubMembers.isActive, true),
    ),
  ) // ← filtered to active
  .limit(1);

if (!foundMember && !orgRole) throw NO_MEMBERSHIP; // ← !orgRole short-circuited

const appRole = foundMember ? foundMember.role : mapClerkRoleToAppRole(orgRole!); // ← fell back to Clerk's stale role
```

For a freshly-deactivated admin, `foundMember` is `null` (because the `isActive` filter excludes them), `orgRole` is still `'org:admin'`, so the `&& !orgRole` guard was false and the code happily granted `'club_admin'` from the JWT. They could then call any admin route — `finances:*`, `riders:create`, even rebooking themselves into the same `club_members` row — until their JWT expired (typically 5–60 minutes depending on Clerk session config).

**Concrete attack:** Club has two admins, A and B. B revokes A's membership. A keeps their existing browser tab open and continues running admin operations until the JWT TTL elapses. Audit-log entries are written under A's `actor_member_id` (which still resolves to the deactivated row) but the writes succeed.

**How I fixed it:**

Defense-in-depth at the tenant resolver. `getTenantContext` now refuses **any** orgId-paired session that lacks an active `club_members` row, regardless of Clerk's `org_role`. `mapClerkRoleToAppRole` is no longer used here — role authority is exclusively `club_members.role`. The `orgRole` JWT claim is dropped from the destructure since nothing in the resolver reads it.

```ts
if (!foundMember) {
  throw new TenantError('NO_MEMBERSHIP', '...');
}
return { ..., orgRole: foundMember.role };
```

The same fix neutralises a second scenario: a Clerk webhook lag that previously masked a missing `club_members` row by letting the user through with the JWT role. Both scenarios collapse into one safe outcome.

**Follow-up flagged (NOT fixed in this pass):** also call `clerk.users.removeOrganizationMembership({ userId: clerkUserId, organizationId: clubs.clerkOrgId })` inside `deactivateMember` so the JWT itself stops carrying the stale role. The defense-in-depth fix above closes the window unconditionally; the structural follow-up is the cleaner long-term shape.

## HIGH — clinical-data exposure within a tenant

### 2. Coaches and grooms could read every horse's medical history (and medication logs)

**Files:**

- `apps/web/app/api/v1/horses/[horseId]/health/route.ts:18-32` (GET)
- `apps/web/app/api/v1/horses/[horseId]/medications/route.ts:16-31` (GET)
- `apps/web/app/api/v1/horses/[horseId]/medications/[medicationId]/logs/route.ts:20-34` (GET)

**What was wrong:** all three GETs gated on `requiredPermission: 'horses:read'`. Looking at `apps/web/lib/permissions-shared.ts`:

- `coach: ['horses:read', ...]` — broad
- `groom: ['horses:read', ...]` — broad
- `veterinarian: ['horses:read', 'horses:read_medical', 'horses:update_medical']` — proper clinical role
- `owner: ['horses:read_own']` — locked OUT of own horse's records (separate UX-fail-closed bug)

So a groom (or any coach) could iterate `/horses/{id}/health` for every horse in the club and exfiltrate surgery history, lameness records, drug logs. The corresponding POST handlers correctly guarded behind `horses:update OR horses:update_medical` (see comment block at `health/route.ts:38-44` explaining the F-5 rationale) — the GET side simply missed the same gate.

**Why it slipped:** these tables hold the `notes` / `skip_reason` / `diagnosis` columns that are encrypted at rest under `MEDICATION_ENCRYPTED_FIELDS` / `MEDICATION_LOG_ENCRYPTED_FIELDS` / `HEALTH_RECORD_ENCRYPTED_FIELDS` (audit F-2 closure). Encryption-at-rest is irrelevant if the route gate that reads them is open to anyone with `horses:read`.

**How I fixed it:**

Inline OR-gate on each GET, mirroring the existing POST pattern:

```ts
const allowed =
  hasPermission(ctx.orgRole, 'horses:read_medical') ||
  hasPermission(ctx.orgRole, 'horses:update_medical') ||
  hasPermission(ctx.orgRole, 'horses:update');
if (!allowed) return errorResponse('FORBIDDEN', '…', 403);
```

For `medications/[medicationId]/logs/route.ts` GET I additionally accepted `horses:update_care` (which grooms have) because grooms doing dose administration legitimately need to see what they've already given today. The other two routes (health, medication-list) stay strict — clinical roles only. Inline comment explains: if a future workflow needs a non-care non-clinical reader, the right answer is granting `horses:read_medical` to that role rather than relaxing the gate.

The `requiredPermission: 'horses:read'` decoration is left in place so the wrapper still does its baseline filter — the inline check is the additional gate, not a replacement.

## MED — money-handling defense-in-depth

### 3. Webhook amount comparison silently passed when event currency was missing

**File:** `apps/web/lib/payments/webhook-helpers.ts:793-823` (booking path) + `:991-1029` (livery path).

**What was wrong:** the existing currency-mismatch guard was

```ts
if (event.currency && event.currency.toUpperCase() !== bookingRef.currency.toUpperCase()) {
  return /* error */;
}
if (event.amountReceivedMinorUnits < bookingRef.amount) {
  return /* underfund error */;
}
```

The `event.currency &&` short-circuit silently skipped the mismatch guard whenever the event lacked a currency field. Then the integer comparison at the next line ran across whatever currency the provider actually charged. Concrete failing case: a Stripe Connect intent in USD without a currency on the parent (some `charge.refund.updated` events have this shape; N-Genius `extractOrderFields` returns `amountCurrency: undefined` whenever the embedded `payment[0].amount.currencyCode` is omitted) silently passed the underfund check whenever the USD minor units happened to fall above the AED fils expected.

**How I fixed it:**

Added an explicit `if (!event.currency)` branch BEFORE the mismatch check, on both the booking and livery paths. New log tag `webhook_amount_without_currency` / `livery_webhook_amount_without_currency` so operator triage can spot the case. Returns `{ kind: 'matched', ... }` (no payment applied) — fail-closed. The original currency-mismatch and amount-underfund branches now only run with a present currency.

## MED — soft-delete defense-in-depth

### 4. Six routes resolved `clubs.timezone` without `isNull(deleted_at)`

**Files:**

- `apps/web/app/api/v1/competitions/route.ts:71-77`
- `apps/web/app/api/v1/competitions/[competitionId]/route.ts:67-74`
- `apps/web/app/api/v1/finances/coupons/route.ts:26-31`
- `apps/web/app/api/v1/finances/coupons/[couponId]/route.ts:20-25`
- `apps/web/app/api/v1/booking-slots/route.ts:83-89`
- `apps/web/app/api/v1/booking-slots/bulk/route.ts:22-28`

**What was wrong:** each route did `db.select({ timezone }).from(clubs).where(eq(clubs.id, ctx.clubId))` inline, missing the `isNull(clubs.deletedAt)` gate. After Clerk's `org.deleted` webhook lands and tombstones the club, these reads would still resolve a deleted club's timezone. Inconsistent with `getClubById` (`packages/db/src/queries/clubs.ts:10`) and the dashboard read paths, all of which gate correctly. Low real-world impact (auth presumably blocks the request before this point — but not guaranteed in race windows).

**How I fixed it:**

Added a `getClubTimezone(clubId)` helper to `packages/db/src/queries/clubs.ts` that bakes in the `isNull(deletedAt)` gate and projects only the timezone column. All six routes now call the helper instead of inlining the query. As a side benefit, the inline `db` + `clubs` + `eq` imports drop from those routes.

### B. Auth-agent #3 — parents cannot read or cancel their child's booking by ID

**Files:**

- `apps/web/app/api/v1/bookings/[bookingId]/route.ts:55` (GET)
- `apps/web/app/api/v1/bookings/[bookingId]/route.ts:105` (DELETE / cancel)
- `apps/web/app/api/v1/bookings/[bookingId]/cancel-preview/route.ts:42`

**What's wrong:** the row-level check accepts `ctx.memberId === booking.riderMemberId`. For parents, the booking's `riderMemberId` is the **child**, not the parent — so parents are silently denied legitimate access. The list endpoint at `bookings/route.ts:91-101` and the payment endpoint at `bookings/[bookingId]/payment/route.ts:91-100` and the competition-entries detail route DO call `isParentOf`. The three above don't. **Fail-closed bug, not a security bypass** — but it's a real UX defect.

**Recommended fix:** add `const isGuardian = canReadChild && await isParentOf(ctx.clubId, ctx.memberId, booking.riderMemberId)` and accept `isSelf || isGuardian`. Mirror the bookings-list pattern.

### C. Money-agent M-1 — coupons have no currency

**Schema:** `packages/db/src/schema/packages.ts:95-128` defines the `coupons` table with no `currency` column. `lesson_types.currency` and `bookings.currency` are per-row, defaulting to AED but free-form.

When a club has lesson types in two currencies, an AED-tuned coupon (`discountType='fixed', discountValue=20000` meaning 200 AED) applied to a USD lesson silently treats it as 200 USD off — roughly a 4× over-discount. `minimumAmount` and `maxDiscount` inherit the same currency-confusion. `validateCoupon` (`packages/db/src/queries/finances.ts:418-531`) receives `params.currency` but only uses it for the error message — it never gates the math.

**Recommended fix:** add `currency` column to `coupons`, default to club's currency at create time, refuse to apply when `coupon.currency !== booking.currency`. Migration + Zod schema update + UI tweak. Latent today because most data is single-currency, but the schema permits the mix.

### D. Money-agent M-2 — `getFinanceOverview` SUMs across currencies

**Files:** `packages/db/src/queries/finances.ts:31, 42, 49, 54, 67, 79`; `packages/db/src/queries/reports.ts:28`. SUMs `bookings.amount`, `livery_invoices.amountMinorUnits`, `expenses.amount` ignoring per-row `currency`. As soon as a club uses two currencies, the dashboard cards lie.

**Recommended fix:** group SUMs by currency, return per-currency totals, render side-by-side in the dashboard. Or enforce single-currency-per-club at the schema level (CHECK constraint plus settings UI removing the per-row override).

### E. Soft-delete-agent — `softDeleteHorse` doesn't cascade to medical children

**File:** `packages/db/src/queries/horses.ts:437-462`. Cascades to `horsePairingHistory` only. `horse_health_records`, `horse_medications`, `horse_documents`, `horse_medication_logs` stay live (read-blocked via `isHorseActiveInClub` so no leak today, but):

- GDPR right-to-erasure not honoured for medical records.
- A future query that forgets `isHorseActiveInClub` would leak.
- A hard-purge later silently CASCADE-deletes the children unexpectedly.

**Recommended fix:** explicit deletes (or soft-delete columns) on the four child tables, mirroring the `horsePairingHistory` cascade. Also wrap the cascade in the same transaction as the parent UPDATE.

### F. Templates-agent — `<Img src={clubLogo}>` not gated through `safeHref`

**Files:** 5 templates render `<Img src={clubLogo}>` without scheme-checking. Today `clubs.logo_url` is only written from Clerk's `org.image_url` (trusted). If a future "edit club logo" UI ships and routes through the existing zod `nullableOptionalUrl` schema (which accepts any URL-parseable scheme), `data:` images would render — usable as a tracking-pixel leak vector to email recipients.

**Recommended fix:** add an `<Img>`-side `safeImgSrc(url)` helper that allowlists `https://` + `data:image/{png,jpeg,webp,gif};base64,` and returns a transparent 1×1 fallback otherwise. Or scheme-restrict the schema upstream.

### G. Money-agent L-2, S-1, S-2

- **L-2:** `coerceFeePercent` allows non-integer percent through `Number(value)` — `Math.round` of `25.5%` is merchant-favoring on .5-cent boundaries. Off-by-1 fil at most. Decide whether the fee admin form should constrain to integers.
- **S-1:** Refund route uses pre-lock `booking.amount` (line 200) inside the locked transaction. Every other monetary field is re-read from `locked`. If a future writer mutates `bookings.amount` post-creation, the locked refund math reads stale.
- **S-2:** Refund cap minus cancellation fee leaves `paymentStatus='partial'` permanently after cancel. UI may misrepresent this as "money still owed."

### H. Auth-agent #4-#6

- **#4:** Coaches see full rider roster including decrypted medical notes + emergency contacts. Confirm with product whether coaches should see medical for riders they don't actually coach.
- **#5:** Dead permissions in the matrix (`coach.bookings:update_own`, `coach.riders:update_notes` — no route checks them).
- **#6:** Comment-cementing the role-precedence invariant in `tenant.ts` so a future refactor can't accidentally reverse `foundMember.role` vs `orgRole`.
