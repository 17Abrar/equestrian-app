## HIGH — money-misroute / wrong-amount-charged

### 1. Booking-payment idempotency key didn't include amount → Stripe replays old amount on retry

**File:** `apps/web/app/api/v1/bookings/[bookingId]/payment/route.ts:232` (was).

**What was wrong:** the idempotency key was `booking_${booking.id}` — a single value per booking.

Stripe (and Ziina, and N-Genius) cache responses keyed on idempotency key and **replay the original response body verbatim** for the same key. So if a booking's amount changed between two payment-init calls (admin applies a coupon, lesson-type price gets adjusted, package-credit changes the math), the second call:

1. Sends the new `amountMinorUnits: 5000` to Stripe.
2. Stripe sees the cached `booking_abc-123` response body and returns it — with `amount: 7500` from the original call.
3. The rider sees and pays the **stale** amount.

This is exactly the failure mode Stripe documents for idempotency-key reuse on a mutated state.

**How I fixed it:**

Idempotency key now includes `currency` and `amount`:

```ts
idempotencyKey: `booking_${booking.id}_${booking.currency}_${booking.amount}`,
```

Any business-state change mints a fresh intent. The same fix applies across all three providers because every adapter consumes the same `paymentInput.idempotencyKey` field. Network-retry replays of the SAME logical state still dedup correctly (key is unchanged when amount is unchanged).

## HIGH — performance hazard at scale

### 2. Livery-billing cron N+1: `findHorseBillingAnchor` per horse × up to 1000

**Files:**
- `apps/web/app/api/cron/livery-billing/route.ts:284` (caller)
- `packages/db/src/queries/livery-invoices.ts:65-118` (LEFT JOIN provides the same data)

**What was wrong:** the cron loop body did `await findHorseBillingAnchor(horse.clubId, horse.horseId)` once per horse. At Neon HTTP ~30ms/call × 1000 horses = ~30s of wallclock for the lookup alone, before any provider round-trip. Cloudflare Workers' wallclock budget is ~30s — at ~300+ horses the cron started running into the cap.

The kicker: `findHorsesDueForBilling` already LEFT-JOINs the latest invoice via `lastInvoiceSub` and projects `lastInvoicePeriodStart` (`livery-invoices.ts:92`). The data was already there in the row; the cron was ignoring it and re-querying. The audit comment at line 45 even acknowledges "each row drives a sequential `findHorseBillingAnchor` round-trip" — flagged but not fixed.

**Subtle semantic catch:** the LEFT JOIN's WHERE filtered `inArray(status, ['pending','paid','overdue'])`, which silently differed from `findHorseBillingAnchor` (returns ANY latest, including `cancelled`). For a horse whose latest invoice was `cancelled`:
- Old cron → anchor is the cancelled period → next billing = cancelled period + 1 month → operator's cancel respected.
- Naive swap → anchor would be the previous non-cancelled period → next billing = the cancelled period itself → operator's cancel **silently undone next cron run**.

**How I fixed it:**

Removed the `inArray(status, ...)` filter from the LEFT JOIN so `lastInvoicePeriodStart` matches `findHorseBillingAnchor`'s semantics exactly. Then swapped the cron's call:

```ts
const anchor = horse.lastInvoicePeriodStart
  ? { periodStart: horse.lastInvoicePeriodStart }
  : null;
const period = nextBillingPeriod(horse, anchor, clubToday);
```

Tightened `nextBillingPeriod`'s anchor-parameter type to `{ periodStart: string } | null` (it never read `periodEnd`). Dropped the now-unused `findHorseBillingAnchor` import. The function itself stays exported in case a future caller needs the row's id/status.

Net effect: one DB round-trip per horse drops to zero. The cron now scales linearly with provider+email round-trips only.

## MED — mobile fixes

### 3. Home screen UTC-vs-local-date filter (M-1)

**File:** `apps/mobile/app/(tabs)/index.tsx:111-112`.

Same bug pattern I fixed in `book.tsx` during pass 1, but the home screen's `upcomingBookings` filter was missed. `new Date().toISOString().split('T')[0]` returns the UTC date; in Dubai (UTC+4) at 02:00 local, `today` resolves to yesterday-UTC and the filter classifies yesterday-local lessons as "upcoming" (and today-local as past). Fixed by computing today from local fields, mirroring the `book.tsx:35-43` `toDateString` helper.

### 4. `useCancelBooking` silently dropped the cancellation reason (M-2)

**Files:** `apps/mobile/hooks/use-bookings.ts:97-99`, `packages/api-client/src/client.ts:300-301`.

**What was wrong:** the mutation destructured `{ bookingId, reason }`, then renamed `reason` to `_reason` and passed only `bookingId` to `api.delete`. The server's DELETE expects `cancelBookingSchema`-typed body (`apps/web/app/api/v1/bookings/[bookingId]/route.ts:89` `parseRequiredBody`); without it, every mobile cancel landed with `reason=undefined`. Audit-trail rows wrote `cancellationReason: null`, and any future flow that branches on reason (refund-or-not, late-cancel-fee) lost the signal.

**Why it was structured that way:** the api-client's `delete<T>(path, options?)` had no body slot — it only accepted options. RFC 9110 permits a DELETE body and Cloudflare Workers + Next.js both honor it; the api-client just hadn't exposed the parameter.

**How I fixed it:**

Extended `client.ts` `delete` signature to `delete<T>(path, body?, options?)` — matches the shape of `post`/`put`/`patch`. Then wired the reason through:

```ts
api.delete<Booking>(`/api/v1/bookings/${bookingId}`, { reason })
```

Verified: only one caller of `api.delete` exists (the one being fixed), so the signature change introduces no breakage.

### 5. `ClerkProvider` crashed late on missing env var (M-3)

**File:** `apps/mobile/app/_layout.tsx:19`.

`process.env.EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY!` — the non-null assertion let `<ClerkProvider>` mount with an empty string in a release build where the env var was missing, surfacing a generic Clerk internal error far from the cause. Now throws at import time with a clear message pointing at the env var, mirroring the `EXPO_PUBLIC_API_URL` guard pattern at `lib/api.ts:11-15`.

## MED — outbound resilience + cron drift

### 6. `OPERATOR_ACTIONABLE_PAY_ERROR_CODES` drift across crons (F-73)

**Files:**
- `apps/web/lib/payments/types.ts` (new shared constant + helper)
- `apps/web/app/api/cron/livery-billing/route.ts:23-28` (was inlining the set)
- `apps/web/app/api/cron/platform-billing/route.ts:262, 391` (was inlining only 2 codes)

**What was wrong:** livery-billing checked 4 codes for "operator-actionable, escalate to error log so it pages": `ACCOUNT_NOT_CONNECTED`, `AUTH_FAILED`, `PROVIDER_NOT_CONFIGURED`, `MISSING_CREDENTIALS`. Platform-billing inlined a check for only 2: `PROVIDER_NOT_CONFIGURED` and `AUTH_FAILED`. So a `MISSING_CREDENTIALS` or `ACCOUNT_NOT_CONNECTED` from the platform-billing cron logged at WARN, never paged the operator, and the platform's billing for that pass silently produced invoices without pay links.

**How I fixed it:**

Extracted the canonical set + an `isOperatorActionablePayErrorCode(code)` helper to `apps/web/lib/payments/types.ts`. Both crons now consume the helper. Future codes added to the canonical set propagate everywhere.

### 7. `findClubsWithTrialEndingOn` had no `.limit()` (P-9)

**File:** `packages/db/src/queries/platform-billing.ts:521-549`.

Defensive cap. Every other cron candidate query is bounded (200 / 500 / 1000); this one was the lone exception. Filtered set is small today (a handful of trialing clubs hitting each window), but the bounding pattern is the contract — added `.limit(500)` matching the platform-billing sibling.

## Findings flagged but NOT fixed in this pass

These are real but each is more than a one-PR diff. Listed in priority order.

### A. Cron `await` chains on Resend (P-3 + F-67/F-68)

**Performance hazard + resilience hazard combined.**

Three crons issue emails sequentially with `await`:
- `booking-reminders/route.ts:228` — up to 500 candidates × ~250 ms = 125 s wallclock.
- `horse-care-reminders/route.ts:227` — 4 query types × 500 each = 2000 candidates × ~250 ms = 8+ minutes.
- `livery-billing/route.ts:504` — 200 reminders × ~250 ms = 50 s.
- `platform-billing/route.ts:424, 542` — same shape.

All of them blow Workers' 30 s budget at moderate scale. The livery-billing **issuance** path correctly uses `sendTriggeredEmailAsync` (fire-and-forget); the reminder + care paths should mirror that pattern.

Coupled with **F-67 / F-68** — the issuance crons don't wrap `createPayIntent` in `withProviderRetry`, so a single transient Stripe / Ziina / N-Genius 5xx during a daily run produces invoices without pay links for every horse/club issued in that window.

**Recommended PR:** `sendTriggeredEmailAsync` for reminder paths; wrap all cron-side adapter calls in `withProviderRetry` with the existing `OPERATOR_ACTIONABLE_PAY_ERROR_CODES` set as the no-retry classification.

### B. Platform-billing cron N+1 (P-2)

**File:** `apps/web/app/api/cron/platform-billing/route.ts:126`. `findClubBillingAnchor(club.clubId)` called per-club inside the loop — same shape as P-1 but `findClubsDueForBilling` doesn't pre-join the anchor today. Easy fix: extend the query with the same `last_invoice_sub` LEFT JOIN shape used in `findHorsesDueForBilling`. Bigger than P-1 because the join needs writing.

### E. Missing indexes (P-5, P-6, P-7)

### F. No fetch timeouts on adapter calls (F-70)

11 `fetch` calls in `n-genius.ts`, `ziina.ts`, `platform-ziina.ts` with no `signal: AbortSignal.timeout(N)`. Cloudflare Workers' default fetch lets a hung provider connection burn the entire 30 s isolate budget. Add `AbortSignal.timeout(15_000)` (or similar — 15 s is the conventional payment-gateway upper bound).

### G. Stripe SDK no `timeout` / `maxNetworkRetries` (F-71)

`stripe.ts:95` constructs the SDK with defaults: 80 s timeout, 0 retries. The 80 s exceeds the Worker wallclock. Pass `{ timeout: 15_000, maxNetworkRetries: 0 }` — let our `withProviderRetry` wrapper handle retries.

### H. Mobile API client has no request timeout (M-6)

`packages/api-client/src/client.ts` plumbs `signal` but no mobile hook sets one. On a degraded cell connection, requests hang until the OS kills the socket (iOS ~75 s, Android varies). Add `AbortSignal.timeout(15_000)` as the api-client default.

### I. Resend rate-limit doesn't honor `Retry-After` (F-74)

`lib/email.ts:111-144` retries on fixed `[500, 1500]` ms backoff regardless of what Resend returned in `Retry-After`. On a burst (cron fanning out), retries hit faster than the rate-limit clears, blowing the daily quota.
