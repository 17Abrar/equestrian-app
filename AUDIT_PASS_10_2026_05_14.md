## Findings

### F-1 — `useArenas` and `useLessonTypes` query keys lack list/detail discriminators [MED]

**Where:** `apps/web/hooks/use-bookings.ts:49`, `apps/web/hooks/use-bookings.ts:104`

**Finding:** Both hooks use bare keys (`['arenas']`, `['lessonTypes']`) without the `'list'`/`'detail'` discriminator established by audit pass 9 for `use-horses.ts`/`use-bookings.ts`. Mutations invalidate at the bare prefix:

```typescript
export function useArenas() {
  return useQuery({
    queryKey: ['arenas'],
    queryFn: () => fetchJson<PaginatedResponse<Arena>>(`/api/v1/arenas?pageSize=${MAX_PAGE_SIZE}`),
  });
}
```

**Why it matters:** A future detail-by-id hook (`useArena(arenaId)`) would collide with the list key; mutations that invalidate `['arenas']` would evict detail entries unnecessarily. Locks in a footgun before it's stepped on.

**Recommendation:** Switch to `['arenas', 'list']` and `['lessonTypes', 'list']`; reserve `['arenas', 'detail', arenaId]` / `['lessonTypes', 'detail', id]` for future detail hooks. Update existing invalidations at lines 65, 81, 121, 138, 151 to match the new prefix.

### F-2 — `useRiders` list/detail query keys not split [MED]

**Where:** `apps/web/hooks/use-riders.ts:25`, `apps/web/hooks/use-riders.ts:32`, mutation invalidation `apps/web/hooks/use-riders.ts:49`

**Finding:** `useRiders` (list) uses `['riders', filters]` and `useRider` (detail) uses `['riders', riderId]`. `useUpdateRider` invalidates at the broad prefix `['riders']`, which over-invalidates detail entries on every list mutation:

```typescript
return useQuery({
  queryKey: ['riders', filters],  // list
  ...
});

return useQuery({
  queryKey: ['riders', riderId],  // detail
  ...
});

onSuccess: () => {
  void queryClient.invalidateQueries({ queryKey: ['riders'] });  // evicts BOTH
  void queryClient.invalidateQueries({ queryKey: ['riders', riderId] });
}
```

**Why it matters:** Detail views refetch unnecessarily on every list mutation, adding RTT to the rider profile page. Also, the array-vs-object position discriminates the two by shape only, which is brittle.

**Recommendation:** `['riders', 'list', filters]` and `['riders', 'detail', riderId]`; invalidations split into `['riders', 'list']` (list mutations) and `['riders', 'detail', riderId]` (per-row updates).

### F-3 — `useCreateRider` response typed as `ApiResponse<unknown>` [LOW]

**Where:** `apps/web/hooks/use-riders.ts:60`

**Finding:** The mutation erases the response shape:

```typescript
return useMutation({
  mutationFn: (data: CreateRiderInput) =>
    fetchJson<ApiResponse<unknown>>('/api/v1/riders', { ... }),
  ...
});
```

**Why it matters:** Consumers receive `unknown` and must cast or trust the shape, breaking the typed-response contract established for sibling mutations (`useCreateStaff`, `useCreateOwner`). If the server-side projection drifts, no compile-time error.

**Recommendation:** `fetchJson<ApiResponse<Rider>>` — the endpoint returns a full `Rider` per the route handler. Mirrors the pattern in `useUpdateRider` at line 42.

### F-4 — `useRiders` query key includes raw filter object [LOW]

**Where:** `apps/web/hooks/use-riders.ts:25`

**Finding:** `queryKey: ['riders', filters]` includes the entire `filters` object verbatim. Filter type is `Partial<RiderFiltersInput>`, so callers can pass unknown keys that would inflate the cache.

```typescript
export function useRiders(filters: Partial<RiderFiltersInput> = {}) {
  ...
  return useQuery({
    queryKey: ['riders', filters],
    ...
  });
}
```

**Why it matters:** Two semantically identical queries with extra throwaway keys would miss cache hits.

**Recommendation:** Build a normalized key object using only the fields actually used by the URL params: `{ search, skillLevel, page, pageSize }`. Pairs naturally with F-2's list/detail split.

### F-6 — Cron schedule drift detection one-sided [LOW]

**Where:** `apps/web/worker-entry.mjs:174-181`

**Finding:** The `scheduled()` handler logs `cron_scheduled_unknown_schedule` when a Cloudflare schedule fires that isn't in `KNOWN_CRON_TARGETS`. But there's no inverse check — if a target is present in the map but no schedule in `wrangler.jsonc:triggers.crons` calls it, the route is silently dead.

```javascript
const KNOWN_CRON_TARGETS = {
  '0 2 * * *': [...],
  '15 2 * * *': [...],
  '0 * * * *': [...],
  '0 3 * * *': [...],
  '30 3 * * *': [...],
};
const cronTargets = KNOWN_CRON_TARGETS[event.cron];
if (!cronTargets) {
  console.error('cron_scheduled_unknown_schedule', { ... });
  return;
}
```

**Why it matters:** A critical billing or reminder cron could be removed from wrangler config but left wired in the targets map. The forward check would never trigger because the schedule never fires. Defense-in-depth gap, not an active outage.

**Recommendation:** Add a startup symmetry check in `instrumentation.ts` that verifies every key in `KNOWN_CRON_TARGETS` is also present in `wrangler.jsonc:triggers.crons` (the wrangler value is available at build time and can be baked in as `process.env.WRANGLER_CRON_SCHEDULES`). Emit a startup-error log if asymmetric.

### F-7 — Discover endpoint pagination hardcoded vs shared constants [LOW]

**Where:** `apps/web/app/api/v1/discover/clubs/route.ts:19-20`

**Finding:** The discover endpoint defines its own pagination cap inline:

```typescript
pageSize: z.coerce.number().int().min(1).max(100).default(20),
```

The shared `paginationSchema` enforces `MAX_PAGE_SIZE = 50` / `DEFAULT_PAGE_SIZE = 25`, capped to avoid blowing the Worker subrequest budget on join-heavy lists.

**Why it matters:** A public endpoint with a higher cap than the shared limit silently diverges. If MAX_PAGE_SIZE is tightened later for subrequest budget reasons, this endpoint won't get the change.

**Recommendation:** Import `paginationSchema` from `@equestrian/shared/schemas` and reuse. If discover specifically needs different limits (e.g., smaller for scrape defense), document it with a comment and define a `DISCOVER_PAGE_SIZE` constant in `packages/shared/src/constants/`.

### F-8 — `useCancelPreview` query key caches `null` bookingId [NIT]

**Where:** `apps/web/hooks/use-bookings.ts:318-322`

**Finding:** Hook accepts `bookingId: string | null` and uses `queryKey: ['cancelPreview', bookingId]`. Query is gated by `enabled: !!bookingId`, so the network call is correctly suppressed, but the cache can hold a `['cancelPreview', null]` key:

```typescript
return useQuery({
  queryKey: ['cancelPreview', bookingId],
  ...
  enabled: !!bookingId,
});
```

**Why it matters:** Cache-key hygiene only — no functional bug.

**Recommendation:** `queryKey: ['cancelPreview', bookingId ?? 'pending']` (or omit the null branch entirely from the hook signature if callers can guarantee a value).

### F-10 — `NotificationPreferences` interface uses snake_case keys [NIT]

**Where:** `packages/shared/src/types/responses/settings.ts:7-25`

**Finding:** The DTO mirrors database column names instead of the camelCase convention used in sibling response types (`ClubSettings`, `Horse`, `Booking`):

```typescript
export interface NotificationPreferences {
  booking_confirmation?: { email: boolean };
  booking_reminder_24h?: { email: boolean };
  booking_cancellation?: { email: boolean };
  // …
}
```

**Why it matters:** Readability and consistency. Developers context-switch from `advanceBookingDays` (camel) to `booking_confirmation` (snake) within the same domain.

**Recommendation:** Either rename the type's keys to camelCase + add a transform in `updateNotificationsSchema` to/from snake_case for the JSON wire format, OR add an inline comment documenting the deliberate exception (the DB jsonb column shape is the actual on-wire format; renaming requires a backfill).
