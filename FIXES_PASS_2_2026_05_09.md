### B. CONFIRMED PHI gaps — encryption-at-rest holes

#### B-1. `rider_profiles.emergencyContactName / emergencyContactPhone / emergencyContactRelation` plaintext at rest

**Schema:** `packages/db/src/schema/rider-profiles.ts:21-23`. **Reads/writes:** `packages/db/src/queries/riders.ts:99-101, 141-143, 180-182, 249-251, 323-330, 349-351`.

The `medicalNotes` column on the same row is encrypted (audit F-2 closure); the emergency contact columns sit literally on adjacent lines, plaintext. An emergency-contact phone + relation (`"Mum, +971 50 ..."`) is stronger PII than the medical notes themselves and lives inside the same audit blast radius (a Neon backup leak, a Drizzle Studio session). Logger and now the audit-log scrubber catch it on the way out (#3 above), but the bytes on disk are cleartext.

**Recommended fix:** add `RIDER_PROFILE_ENCRYPTED_FIELDS = ['emergencyContactName', 'emergencyContactPhone', 'emergencyContactRelation']` in `riders.ts`, mirror the `RIDER_MEDICAL_ENCRYPTED_FIELDS` wiring on insert/update/get, ship a backfill migration shaped like `0034_rider_medical_notes_backfill.sql`.

#### B-2. Horse-care reminder cron emails plaintext PHI to Resend

**File:** `apps/web/app/api/cron/horse-care-reminders/route.ts:277` + the candidate loaders in `packages/db/src/queries/horse-health.ts:732-832, 881+`.

The reminder email body interpolates `horseMedications.medicationName + dosage`, `horseHealthRecords.title`, `horses.vetName`, `horses.insurancePolicyNumber` directly. Resend stores email content cleartext in their logs. The encryption-at-rest invariant on horse-health rows is undone the moment the cron sends.

**Recommended fix:** redact the email body to a generic ("A medication ends tomorrow — open the app for details") and link recipients into the dashboard. Avoid putting clinical content in third-party-vendor mail logs. Same pattern is correct for the SMS path if/when that lands.

#### B-3. `horseMedications.prescribedBy` (vet name) plaintext

**File:** `packages/db/src/queries/horse-health.ts:243`, schema `horse-health.ts:86`. The current `MEDICATION_ENCRYPTED_FIELDS = ['notes']` doesn't cover `prescribedBy`, which is HIPAA-style "treating provider" identifying information. Add to the encryption set + backfill.

#### B-4. `bookings.coachNotes` plaintext

**File:** schema `bookings.ts:260`, query `bookings.ts:461`. Coaches use this freeform field for post-lesson observations like "rider's asthma flared", "saddle sore after fall." Not in any encryption list, not in `PHI_KEYS`, doesn't substring-match `notes` because the camelCase stem is `coachNotes`. Encrypt + add to PHI_KEYS.

#### B-5. `horseDocuments.description` plaintext on `medical`/`vet`/`xray` categorised docs

**File:** schema `horse-health.ts:298`, query `horse-health.ts:646`. User-typed description on a vet-document upload is PHI. Encrypt under a new `HORSE_DOCUMENT_ENCRYPTED_FIELDS = ['description']`.

#### B-6. `horses.markings`, `gearNotes`, `notes`, `insuranceCoverage` plaintext freeform

**File:** `horses.ts:48, 78, 83, 101`. `markings` reliably collects health-history ("scar on left flank from colic surgery"); `notes` is a catch-all where owners drop chronic-condition notes. At minimum encrypt `markings` and `notes`.

### C. Race-condition + cron findings

#### C-1. Trial-ending nudges have no per-club dedup → cron retry double-sends

**Files:** `apps/web/app/api/cron/platform-billing/route.ts:489-558` (`sendTrialEndingNudges`) + `packages/db/src/queries/platform-billing.ts:510-537` (`findClubsWithTrialEndingOn`). Comment at `platform-billing.ts:500-509` admits "we DON'T currently track a `trial_reminder_count` column" and relies on "the date check" for dedup. The cron route returns 500 from the outer `try` on any thrown error (route.ts:89-96); Cloudflare's `scheduled()` retries 5xx from worker-entry. A same-day retry within UTC `today+3` or `today+1` re-fires the nudge for every match.

**Recommended fix:** add a `clubs.trial_reminder_3day_sent_at` + `clubs.trial_reminder_1day_sent_at` (or a small `trial_reminders_sent` JSONB) and CAS-guard the send. Or use the Upstash idempotency key shape from PR #85's `apps/web/lib/idempotency.ts` keyed by `(clubId, threshold-day)`.

#### C-2. Horse-care reminder dedup row written before email send → permanent skip on transient send failure

**File:** `apps/web/app/api/cron/horse-care-reminders/route.ts:204-241`. `recordHorseCareReminderSend` inserts the dedup row first; `sendTriggeredEmail` runs outside the insert's try-rollback. The catch logs but does NOT delete the dedup row, so a Resend hiccup permanently silences the reminder. Booking-reminder cron does the right thing — `unmarkBookingReminderSent` on failure (`apps/web/app/api/cron/booking-reminders/route.ts:264-279`). Mirror that pattern: add `unrecordHorseCareReminderSend` and call it on send failure.

#### C-3. `pruneAuditLog` co-located inside `livery-billing` cron — single point of failure

**File:** `apps/web/app/api/cron/livery-billing/route.ts:90-98`. There's no separate `audit-prune` route, no entry in `wrangler.jsonc` triggers, no entry in `worker-entry.mjs:159-168` `KNOWN_CRON_TARGETS`. If livery-billing 401s (cron-secret rotation drift), worker-entry refuses unknown schedules, or the route's outer try fires, audit-log retention silently freezes. F-61's per-prune try/catch protects prune from issuance/reminder failures *within the same handler*, not from the route itself failing to run.

**Recommended fix:** extract to `apps/web/app/api/cron/audit-prune/route.ts` and add a `0 3 * * *` cron in `wrangler.jsonc` so retention runs even when billing's down.

### D. Webhook hardening

#### D-1. Orphaned-webhook-secret reuse evades the F-33 hash uniqueness check

**File:** `packages/db/src/queries/payment-accounts.ts:252-258`. F-33 enforces "no two CURRENT clubs share the same `webhookSecretHash`" via a partial unique index. But when Club B disconnects (`webhook_secret_hash` set NULL or row deleted), B's hash leaves the table. If Club A then pastes B's old `whsec_…`, A's upsert wins the hash slot. Old Stripe webhooks signed with that old secret would now verify at A's URL.

**Recommended fix:** add a `burned_webhook_secret_hashes (provider, secret_hash, club_id, retired_at)` history table; `connectAccount` checks the hash against both the live table AND the burned table; on disconnect the previous hash moves to burned with a TTL or kept indefinitely. Burned rows are 64-byte hashes — cheap to keep.

#### D-2. N-Genius webhook parses body BEFORE signature verification

**File:** `apps/web/app/api/webhooks/n-genius/route.ts:103`. Unlike Stripe / Ziina (where `verifyWebhook` does parse-after-HMAC), N-Genius needs the body to read `outletId` so it can pick the right secret. Practical impact is moderate: a malformed body crashes the parser before any auth gate, which surfaces as a 400 — but the path before that does run a DB lookup keyed on attacker-controlled `outletId`, opening a slow timing channel between "outlet exists, decrypt happens" and "outlet missing, 401 returned."

**Recommended fix:** if the cost is justified (low-traffic webhook), pre-extract `outletId` with a regex against the raw body string before full `JSON.parse`, then parse-and-verify after the secret is loaded. Or accept the timing channel and document it (current rate limit is 60/min/IP — slow probe at best).

#### D-3. Stripe webhook body cap is "memory-late" not "network-early"

**File:** `apps/web/lib/payments/webhook-body.ts:17-30`. `request.text()` buffers the whole body into memory; the post-read length check fires only after that allocation completes. Cloudflare's per-isolate memory limit is the actual cap (~128 MB on standard Workers), so this isn't unbounded — but a `content-length`-omitted streaming attacker can sustain ~60 large allocations per minute per IP. Defensive; not urgent.

**Recommended fix:** either trust Cloudflare's per-isolate memory cap and document this trade-off, or read the body as a stream with a hard byte cap that aborts mid-read.

### E. Migration set — backfill no-ops + lock-storm hazards (migration-agent)

#### E-1. Three "backfill historical rows" UPDATEs are functional no-ops

**Files:**
- `migrations/0044_audit_r3_notifications_updated_at.sql:18-20`
- `migrations/0047_audit_r5_schema_drift.sql:68-77`
- `migrations/0049_audit_r6_schema_drift.sql:92-95`

Each adds a `NOT NULL DEFAULT now()` column then tries to backfill historical rows by predicate `WHERE updated_at = created_at` (or similar). Postgres runs `ADD COLUMN ... DEFAULT now()` as a single atomic operation, so `updated_at` for every existing row equals the migration's `now()` — not the original event time — and the equality predicate matches zero rows. Mild functional impact (column has *a* value, just `now()` instead of event-truthful). The audit trail comments claiming "backfilled to event time" are inaccurate.

**Recommended fix:** if event-truthful is required, the predicate has to be `WHERE updated_at >= '<deploy-window>'` against a recorded deploy timestamp; otherwise drop the misleading backfill block from the SQL and update the audit comment.

#### E-2. `migrations/0028_misc_indexes_and_constraints.sql:41-42` — `SET NOT NULL` on hot `bookings` table without `lock_timeout`

**File:** `migrations/0028_misc_indexes_and_constraints.sql:41-42`. ACCESS EXCLUSIVE for the duration of the validation pass; on the busiest table in the schema. Already shipped, so this is a "next time" lesson — every future hot-table `SET NOT NULL` / `ALTER COLUMN TYPE` should `SET LOCAL lock_timeout = '5s'` and retry on lock failure.

#### E-3. `migrations/0041_audit_r2_composite_fks.sql:14-20` `SET NOT NULL` after a subquery that can yield NULL

**File:** see header. Comment says community feature is unshipped (zero rows) so it's safe today. If community ever ships before this migration is rerun against a fresh environment with non-empty data, it will abort. Same lesson as E-2 — every backfill that drives a `SET NOT NULL` should `WHERE EXISTS` cleanup or DELETE orphans first, like sibling 0017/0038/0039 do.

#### E-4. `scripts/migrate-neon.mjs:203-209` swallows idempotency errors only for split-statement files

The idempotency error swallowing only matches files split on `--> statement-breakpoint` markers. Files 0006-0051 (46 of 52) don't have those markers and run as a single statement, so any sub-statement failure aborts the whole file before later sub-statements run. Recommended: switch the runner to wrap each migration in an explicit `BEGIN; ... COMMIT;` block (consistent with what `drizzle-kit migrate` does) so partial application can't leave the schema in a frankenstein state.
