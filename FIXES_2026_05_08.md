## CRITICAL — production-blocking bug

### 1. Migration `0048_audit_r6_webhook_secret_hash_uniqueness.sql` was orphaned from the journal

**What was wrong**

When PR Nu-2 (commit `7278d45`) added the migration to enforce per-club uniqueness on the webhook signing secret, the SQL file landed on disk as `0048_audit_r6_webhook_secret_hash_uniqueness.sql` — colliding with the existing `0048_audit_r6_horse_care_phi_encryption.sql` from PR Psi (commit `8cfaac3`) — and the journal entry was missed.

The fail mode is sneaky: **tests pass, production breaks**.

- `packages/db/src/test/harness.ts:26-37` applies migrations by `readdir + sort`-ing the folder. So in tests the orphan file applied fine and the `webhook-secret-uniqueness.test.ts` suite passed (10 of the 92 db tests).

- `packages/db/src/schema/finances.ts:85` already declares the column. `packages/db/src/queries/payment-accounts.ts:222-299` already reads/writes it.
- Concrete prod symptom: every payment-account reconnect (Stripe, Ziina, N-Genius) would 500 with `column "webhook_secret_hash" of relation "club_payment_accounts" does not exist`. F-33 was supposed to be closed by this PR — it wasn't, in production.

**How I fixed it**

- Renamed the file to the next available index: `0048_… → 0051_audit_r6_webhook_secret_hash_uniqueness.sql` (via `git mv`).
- Appended the journal entry at `idx=51`, `when=1777248000051`, `tag` matching the new filename — see `packages/db/migrations/meta/_journal.json`.
- Verified: `cd packages/db && pnpm test` still 92/92 (test harness applies all SQL files alphabetically; the new `0051_*` lands after `0050_n_genius_outlet_unique`, which is fine because the change is purely additive).
- Folder vs. journal alignment script:
  ```
  Journal entries: 52
  SQL files: 52
  Orphan files: NONE
  Journal monotonic: OK
  ```

**Why it matters**

This is the most expensive class of bug — green CI, broken prod. Without this fix the next prod deploy would have left the schema lying about which columns exist.

**Follow-up I noticed but didn't apply**

`packages/db/src/test/harness.ts:14-18` has a stale comment that says hand-written migrations are "absent from the journal" — that's no longer true (the README explicitly requires journaling them). ## HIGH — confirmed concurrency / settlement bugs

### 2. Reminder counter race in livery billing — double-fire of overdue emails

**File:** `packages/db/src/queries/livery-invoices.ts:370` (`markInvoiceOverdueAndLogReminder`) + `apps/web/app/api/cron/livery-billing/route.ts:503-526`

**What was wrong**

`markInvoiceOverdueAndLogReminder` ran `UPDATE … SET reminder_count = reminder_count + 1, last_reminder_at = now() WHERE id = $invoiceId AND club_id = $clubId` with **no CAS predicate** on `reminder_count`. The cron's caller didn't check the return value either.

Cloudflare Workers can replay scheduled() across regions, and `worker-entry.mjs` retries 5xx — so two isolates can run the cron concurrently. Both call `findOverdueInvoicesForReminders`, both observe `reminder_count = 0`, both pass the `daysOverdue ≥ REMINDER_THRESHOLDS[0]` gate, both `setInvoiceProviderRef` (correctly idempotent — Stripe replays the PI), both call `markInvoiceOverdueAndLogReminder` — and Postgres serializes the two UPDATEs, advancing `reminder_count` 0 → 1 → 2 in one pass. Both isolates then fire the day-7 reminder email. Net: owner gets two day-7 nudges and the day-14 nudge gets silently skipped forever (`REMINDER_THRESHOLDS[2] === undefined` for `reminder_count = 2`).

**How I fixed it**

Added a CAS on `reminder_count` matching the value the caller observed. Symmetric to the existing `markBookingReminderSent` CAS at `packages/db/src/queries/bookings.ts:1138-1149`:

```ts
export async function markInvoiceOverdueAndLogReminder(
  clubId: string,
  invoiceId: string,
  expectedReminderCount: number,  // ← new
) {
  const result = await rawDb
    .update(liveryInvoices)
    .set({ status: 'overdue', lastReminderAt: new Date(),
           reminderCount: sql`${liveryInvoices.reminderCount} + 1`,
           updatedAt: new Date() })
    .where(and(
      eq(liveryInvoices.id, invoiceId),
      eq(liveryInvoices.clubId, clubId),
      eq(liveryInvoices.reminderCount, expectedReminderCount),  // ← new
    ))
    .returning();
  return result[0] ?? null;
}
```

Updated the cron caller to pass `inv.reminderCount` and treat a `null` return as "another isolate beat me — skip the email":

```ts
const claimed = await markInvoiceOverdueAndLogReminder(
  inv.clubId, inv.invoiceId, inv.reminderCount,
);
if (!claimed) { skipped += 1; continue; }
```

`findOverdueInvoicesForReminders` already projects `reminderCount` (line 321), so the call site has the value.

**Why it matters**

The 7/14/30-day cadence is a contract with the club's owners. Doubling it at day 7 and skipping day 14 erodes trust quickly, and it's invisible to operators because the cron's structured logs don't differentiate "sent" from "double-sent".

### 4. Platform-Ziina webhook silently swallows paid-event-for-cancelled-invoice

**File:** `apps/web/app/api/webhooks/ziina-platform/route.ts:194-271`

**What was wrong**

When a club admin cancels a platform subscription invoice and the club's card lands a payment a moment later (mid-flight at the payment processor), the webhook handler:

1. Looks up the invoice by `providerPaymentId` → finds it.
2. Reconciles currency / amount → passes (or just logs overfund).
3. Calls `markPlatformInvoicePaid`, which has `inArray(status, ['pending', 'overdue'])` — so a `cancelled` row returns `null`.
4. Falls into the `if (!updated)` branch, logs `platform_webhook_invoice_already_terminal` at INFO level, returns `null`.
5. Caller marks the dedup row as `processed`. Operator gets no signal.
6. Money sits unreconciled in Cavaliq's Ziina balance.

The same shape was caught and fixed for **livery** (`livery_webhook_paid_for_cancelled_invoice` — F-13 in the audit) and for **bookings** (`webhook_payment_for_inactive_booking` — audit AI-24). The platform-billing webhook is the third instance and was missed.

**How I fixed it**

Mirrored the F-13 livery pattern. Before calling `markPlatformInvoicePaid`, branch on `invoice.status === 'cancelled'`:

- If `eventAmountReceived > 0`: log `platform_webhook_paid_for_cancelled_invoice` at ERROR level and return a `permanentFailureReason` string. The caller's existing `permanentFailureReason` plumbing (lines 141-167) marks the dedup row `permanently_failed`, which fires the alert pipeline.
- If no payment in the event: idempotent no-op (existing `already_terminal` log).

`findPlatformInvoiceByProviderPayment` already projects `status` (line 242), so the data is available.

**Why it matters**

This is the path where a club has paid Cavaliq and Cavaliq has the money, but operations doesn't know about it. Three months of those quietly accumulate into a reconciliation nightmare.

## MED — defense-in-depth fixes

### 5. `setBookingPaymentRef` accepts new `providerPaymentId` writes against `cancelled` / `no_show` bookings

**File:** `packages/db/src/queries/bookings.ts:929-994`

**What was wrong**

The function's CAS predicates guarded against `paymentStatus` regression (paid → pending, refunded → anything, etc., per audit F-29 and HIGH-13) — but did NOT guard against the booking's **lifecycle status** being terminal. The payment-init route (`apps/web/app/api/v1/bookings/[bookingId]/payment/route.ts:266`) reads the booking, calls the provider (a real Stripe / Ziina / N-Genius PI is minted), and then writes `providerPaymentId` back. A concurrent admin DELETE that flips the booking to `cancelled` between the read and the write is not blocked by the CAS, so the new PI gets attached to a cancelled booking and the rider's card is charged.

`webhook-helpers.ts` does have a `webhook_payment_for_inactive_booking` branch that catches the eventual `succeeded` event, but it's a downstream cleanup — the rider has already been charged, and reconciliation is manual.

**How I fixed it**

Added a defense-in-depth predicate to the WHERE clause when `providerPaymentId` is being written:

```ts
conditions.push(
  sql`${bookings.status} NOT IN ('cancelled', 'no_show')`,
);
```

This is conservative: it only fires when `data.providerPaymentId` is set (so webhook-side paymentStatus updates on already-cancelled rows still work as before — those have their own upstream lifecycle guard in webhook-helpers). The payment-init's TOCTOU window now closes here instead of slipping through.

A complete fix would also wrap the payment-init route's read-then-call-provider-then-write in `writeTransaction(... SELECT FOR UPDATE)` — that lives in `apps/web/app/api/v1/bookings/[bookingId]/payment/route.ts` and was on the agent's list (race-condition R-4) but is a larger surgical change. The CAS predicate fixes the data-corruption symptom; the FOR UPDATE wrap is the proper structural fix and is flagged below.

## MED — DoS prevention via body-cap migration

### 6. 44 API routes still used `request.json()` directly — migrated to `parseRequiredBody`

**Files:** see git diff — every `apps/web/app/api/v1/**/route.ts` lint warning.

**What was wrong**

`apps/web/lib/api-utils.ts:259-315` exposes `parseOptionalBody` and `parseRequiredBody`, which:

1. Reject requests where `Content-Length` exceeds `MAX_REQUEST_BODY_BYTES` (1 MB).
2. After reading the body, re-check actual length against the cap (defense against an attacker-controlled `Content-Length` header).
3. Map JSON parse errors to a clean `400 INVALID_JSON` instead of bubbling a `SyntaxError` that `withAuth` renders as a `500 INTERNAL_ERROR`.

44 routes still called `await request.json()` directly. Each one was both a DoS vector (no body cap, attacker streams 100 MB into the worker) and a malformed-input → 500 hazard. The author had landed an ESLint `no-restricted-syntax` rule as `warn` (not `error`) at `tooling/eslint/nextjs.js:67-78` because migrating all 50+ in one PR was too high blast radius. F-63 in the audit explicitly listed this as deferred.

**How I fixed it**

Did the migration. Mechanical replacement:

```ts
// before
const body = await request.json();
const data = validateInput(schema, body);

// after
const data = await parseRequiredBody(request, schema);
```

A few non-mechanical cases needed care, all noted in the migration log:

- `apps/web/app/api/v1/me/active-club/route.ts` — bypasses `withAuth` (it changes the active tenant). Added a `PayloadTooLargeError` import + a 413 catch arm to its hand-rolled error handler.
- `apps/web/app/api/v1/settings/route.ts` — composite PATCH that fans out into 5 separate `safeParse` calls. No single Zod schema applies, so I introduced `passthroughObjectSchema = z.object({}).passthrough()` to give the helper a body-cap + JSON-shape guard while preserving the per-section validation.
- `apps/web/app/api/v1/booking-slots/[slotId]/route.ts` — was the only file with two `request.json()` sites (PATCH + DELETE). Both migrated.
- Routes that used `validateInput` for query strings AS WELL AS body kept the import (only the body callsite was migrated).

**Verification**

Lint went from 45 `no-restricted-syntax` warnings to 0. The single remaining warning is unrelated (`console.log` in an e2e test).

## MED — UI bugs

### 7. Search inputs hammer the API on every keystroke

**Files:**
- `apps/web/components/horses/horses-list.tsx`
- `apps/web/components/riders/riders-list.tsx`
- `apps/web/components/owners/owners-list.tsx`
- `apps/web/components/staff/staff-list.tsx`
- `apps/web/components/emails/audiences-tab.tsx` (`LivePreview`)

**What was wrong**

Each list page has a "Search …" input that drives a TanStack Query keyed off the raw input value. Typing "thunderbolt" fires 11 requests against `/api/v1/horses?search=thunderbolt`. The `discover` page already does it correctly with a 250 ms `useEffect` debounce; the four dashboard list components don't.

Worse, `audiences-tab.tsx`'s `LivePreview` had a comment that **claims** to debounce — `useMemo(() => JSON.stringify(filters), [filters])` — but `useMemo` only memoizes the React Query KEY; every filter change still mints a fresh key and immediately fires the queryFn. The `audiences/preview` endpoint is on the heavy side (audit F-5 flagged it as unbounded; that finding is closed in code with a row cap, but it's still expensive), so per-keystroke amplification on it is the worst place to skip debouncing.

**How I fixed it**

Created a shared hook at `apps/web/hooks/use-debounced-value.ts`:

```ts
export function useDebouncedValue<T>(value: T, delayMs = 250): T { … }
```

Wired it into all five components. Each list now keys the query off `useDebouncedValue(search, 250)` instead of the raw input. `LivePreview` debounces the stringified-filters key AND the request body, so neither is computed per keystroke.

### 8. UTC date string used where local date was expected — calendar shows wrong "today"

**Files:**
- `apps/mobile/app/(tabs)/book.tsx:35-37`
- `apps/web/app/rider/book/page.tsx:102-104`
- `apps/web/components/horses/livery-tab.tsx:120`
- `apps/web/components/horses/health-tab.tsx:211, 434`

**What was wrong**

Pattern across these files:

```ts
function toDateString(d: Date): string {
  return d.toISOString().split('T')[0]!;
}
```

`Date.toISOString()` returns the **UTC** date. In Dubai (UTC+4) at 02:00 local time, the local date is May 9 but `toISOString()` returns `2026-05-08`. Effects:

- The booking calendar's "today" cell highlights the wrong day.
- `isPast` boundary (`d < new Date(toDateString(new Date()) + 'T00:00:00')`) flips at the wrong moment — yesterday-local renders as `isPast=false`.
- "Default to today" date inputs in horse health-records and livery-end-date forms default to yesterday for users in any timezone west of UTC during local-morning hours, and to tomorrow east of UTC during local-evening.

**How I fixed it**

Rebuilt the date string from local fields:

```ts
function toDateString(d: Date): string {
  const yyyy = d.getFullYear();
  const mm = String(d.getMonth() + 1).padStart(2, '0');
  const dd = String(d.getDate()).padStart(2, '0');
  return `${yyyy}-${mm}-${dd}`;
}
```

Added a shared `getTodayLocalDateString()` to `packages/shared/src/utils/timezone.ts` (alongside the existing `getTodayDateString(timezone)` server-side helper) and exported it. Wired it into `horses/livery-tab.tsx`, `horses/health-tab.tsx`. The two book pages got an inline version of the same logic (they already had a local `toDateString` helper).

**Why it matters**

The bug shows up most visibly to UAE-based clubs (the primary launch market) at certain times of day. It's the sort of thing a rider would screenshot and send to support as "your app thinks today is yesterday."

## Tenant-scoping audit — clean

The cross-tenant scoping agent walked every `db.select / update / delete / insert` in `packages/db/src/queries/*.ts` (~8.7k lines, 22 files) and every API route, cross-referencing against F-1..F-76. **No CONFIRMED tenant-leak bugs found.** Every query is either club-scoped via `eq(table.clubId, ctx.clubId)`, joins through a parent FK that itself carries `clubId`, or is intentionally tenant-global (webhook claim/process by `(provider, eventId)`, `/discover`).

Two LIKELY/SUSPICIOUS items the agent surfaced — neither is a tenant leak:
