## Fixes Applied

### 1. Web fetch wrapper now has bounded runtime

**File:**
- `apps/web/lib/fetch-json.ts`

**Issue:** The shared web `fetchJson` wrapper used by TanStack Query hooks and form mutations had envelope validation and CSRF headers, but no request timeout. A stalled network/proxy request could leave UI mutations and queries waiting indefinitely.

**Fix:** Added a composed `AbortController` timeout with caller-signal forwarding and a typed `RequestTimeoutError`. Timed-out mutations now surface a deterministic retryable message instead of hanging.

### 2. Admin refunds no longer ledger failed/pending provider refunds

**File:**
- `apps/web/app/api/v1/bookings/[bookingId]/refund/route.ts`
- `apps/web/app/api/v1/bookings/[bookingId]/route.ts`

**Issue:** The standalone refund route and the cancel-with-refund path called the provider and then advanced the local booking refund ledger without first checking `refund.status`. The standalone route's inline audit note already identified the correct fix: use a stable idempotency key and only ledger terminal succeeded refunds.

**Fix:** Removed `Date.now()` from the standalone refund provider idempotency key, added status checks before booking-ledger mutation in both paths, returns `502` when the provider reports `failed`, and avoids ledger mutation for `pending` refunds. The standalone refund route returns `202` for pending provider state. The success response now uses the live locked cancellation-fee value when computing remaining refundable balance.

### 3. Refund UI distinguishes pending from issued

**File:**
- `apps/web/components/bookings/bookings-list.tsx`

**Issue:** The bookings UI always toasted `Refund issued` after the refund route returned success. With provider-pending refunds now represented explicitly, that message would overstate the ledger/payment state.

**Fix:** Added a typed refund response shape and changed the toast to warn that the booking ledger updates after provider confirmation when the route returns `status: "pending"`.

### 4. Freeform reasons are no longer duplicated into logs/audit

**Files:**
- `apps/web/app/api/v1/bookings/[bookingId]/route.ts`
- `apps/web/app/api/v1/booking-slots/[slotId]/route.ts`
- `apps/web/app/api/v1/competitions/[competitionId]/classes/[classId]/entries/[entryId]/route.ts`
- `apps/web/app/api/v1/horses/[horseId]/decline/route.ts`

**Issue:** Several routes stored legitimate business reasons on the target record, but also duplicated those freeform strings into structured logs or audit-log changes. Users can type phone numbers, emails, medical context, or other personal details into these fields, so duplicating them increases retention/exposure without adding operational value.

**Fix:** Removed reason text from logger/audit payloads and kept only non-sensitive operational metadata such as `reasonProvided`. The canonical reason remains on the business record and in customer-facing emails where already intended.

### 5. Clerk organization deletion now closes local tenant access

**Files:**
- `apps/web/app/api/webhooks/clerk/route.ts`
- `apps/web/lib/tenant.ts`
- `apps/web/app/api/v1/me/active-club/route.ts`

**Issue:** A live Clerk production webhook test showed `organization.deleted` arrived and soft-deleted the club, but Clerk did not emit a corresponding `organizationMembership.deleted` event within the verification window. That left the local `club_members.is_active` flag true under a soft-deleted club.

**Fix:** The Clerk `organization.deleted` handler now deactivates local club members for the deleted club. The tenant resolver and active-club switcher also filter out inactive/soft-deleted clubs, so a stale active member row cannot resolve a deleted tenant.
